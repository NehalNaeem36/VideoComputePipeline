# VideoComputePipeline - Windows Implementation Prompt for Codex

You are implementing a C11 project called VideoComputePipeline on Windows.

The user has already installed:
- FFmpeg development libraries and runtime
- OpenCL headers/runtime
- GCC
- CMake
- Make / MinGW make

Assume Windows using MSYS2 UCRT64 or MinGW-style GCC unless the local setup indicates otherwise.

Project goal:
Build a modular CPU/GPU video-processing benchmark pipeline.

The program must:
1. Read an MP4 using FFmpeg libraries, not FFmpeg CLI.
2. Decode video frame by frame.
3. Convert decoded frames into an internal RGB24 Frame structure.
4. Process each frame using either CPU filters or OpenCL GPU kernels.
5. Encode processed RGB24 frames back into an MP4 using FFmpeg libraries.
6. Generate benchmark CSV files comparing CPU and GPU performance.
7. Keep the code clean, modular, and employer-presentable.

Do not use FFmpeg command-line calls.
Do not shell out to ffmpeg.exe.

Use FFmpeg libraries directly:
- libavformat
- libavcodec
- libavutil
- libswscale

Use OpenCL for GPU work.

Use C11.
Use CMake.

Target filters:
- grayscale
- 3x3 box blur
- 5x5 box blur
- 9x9 box blur

The CPU implementation should be direct C.
The GPU implementation should use OpenCL kernels.

The benchmark input video is expected to be around 4 minutes 20 seconds long, but the program should work with any valid MP4 input.

============================================================
HIGH-LEVEL PIPELINE
============================================================

First implementation should be sequential:

decode one frame
    -> process frame on CPU or GPU
    -> encode processed frame
    -> record timing
    -> repeat until end of video

The design should allow a later threaded/interleaved version:

decoder thread
    -> raw frame queue
    -> CPU/GPU processing worker
    -> processed frame queue
    -> ordered writer thread

Important:
- Do not decode the whole video before processing.
- Process frames as they become available.
- Preserve frame.index for ordering and benchmarking.
- The first version may be sequential, but the modules should not block future pipelining.

============================================================
CORE FRAME MODULE
============================================================

Files:
- include/core/frame.h
- src/core/frame.c
- tests/core/frame_test.c

Purpose:
The frame module defines and manages the internal frame representation used across the project.

It must not contain:
- FFmpeg logic
- OpenCL logic
- CPU filter logic
- benchmark logic
- threading logic

It only manages frame memory and metadata.

Define:

typedef enum {
    FRAME_FORMAT_RGB24 = 0,
    FRAME_FORMAT_GRAY8 = 1
} FrameFormat;

typedef struct {
    int index;
    int width;
    int height;
    int channels;
    FrameFormat format;
    size_t stride;
    size_t size;
    uint8_t* data;
} Frame;

Implement:

void frame_init(Frame* frame);

int frame_alloc(Frame* frame, int width, int height, FrameFormat format);

void frame_free(Frame* frame);

int frame_copy(Frame* dst, const Frame* src);

int frame_move(Frame* dst, Frame* src);

int frame_is_valid(const Frame* frame);

size_t frame_calculate_stride(int width, FrameFormat format);

size_t frame_calculate_size(int width, int height, FrameFormat format);

Rules:
- RGB24 has 3 channels.
- GRAY8 has 1 channel.
- stride = width * channels for now.
- size = stride * height.
- frame_copy must deep copy the pixel buffer.
- frame_move must transfer ownership and clear the source.
- frame_free must free data and reset all fields safely.

- allocate/fill internal Frame

- return frames one at a time

    int width;
    int height;
    double fps;

Public functions:

Return convention for video_reader_read_frame:
- 1 means a frame was decoded successfully
- 0 means end of stream
- negative means error

video_writer files:
- include/video/video_writer.h
- src/video/video_writer.c
- tests/video/video_writer_test.c

Purpose:
Use FFmpeg libraries to encode processed RGB24 Frame objects back into an MP4 file.

video_writer should:
- create output context
- create video stream
- create encoder context
- prefer H.264 if available, otherwise use MPEG-4 fallback
- convert RGB24 Frame to YUV420P using sws_scale()
- set PTS correctly
- send frames to encoder using avcodec_send_frame()
- receive packets using avcodec_receive_packet()
- write packets using av_interleaved_write_frame()
- flush encoder at the end

Public functions:

int video_writer_write_frame(VideoWriter* writer, const Frame* frame);

5x5 reads 25 pixels per output pixel.
9x9 reads 81 pixels per output pixel.
